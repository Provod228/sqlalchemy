Каждый сценарий сопровождается замером времени выполнения, что позволяет наглядно оценить влияние различных подходов на производительность.

## Анализ кода и архитектуры

### Модель Author

```python
class Author(Base):
    __tablename__ = 'authors'
    id = Column(Integer, primary_key=True)
    name = Column(String(50), nullable=False)
    email = Column(String(100), unique=True)
    books = relationship("Book", back_populates="author", cascade="all, delete-orphan")
```

**Достоинства:**
- Чёткое определение таблицы через `__tablename__`
- Использование `nullable=False` для обязательных полей
- `unique=True` для email предотвращает дублирование
- `cascade="all, delete-orphan"` обеспечивает автоматическое удаление книг при удалении автора
- `relationship` с `back_populates` создаёт двунаправленную связь

**Замечания:**
- Отсутствует индекс на поле `email` (хотя `unique=True` создаёт индекс автоматически в большинстве СУБД)
- Нет ограничения на длину email (String(100) — разумно, но можно было бы использовать 255)

### Модель Book

```python
class Book(Base):
    __tablename__ = 'books'
    id = Column(Integer, primary_key=True)
    title = Column(String(100), nullable=False)
    price = Column(Float, default=0.0)
    author_id = Column(Integer, ForeignKey('authors.id', ondelete='CASCADE'))
    author = relationship("Author", back_populates="books")
```

**Достоинства:**
- `ForeignKey` с `ondelete='CASCADE'` обеспечивает целостность на уровне БД
- `default=0.0` для цены — разумное значение по умолчанию
- `relationship` правильно настроена для доступа к автору из книги

**Замечания:**
- `price` имеет тип `Float`, что может привести к проблемам с точностью для денежных сумм. В реальном проекте стоило бы использовать `Numeric(10, 2)`
- Отсутствует проверка, что цена не может быть отрицательной (хотя это можно было бы добавить через `CheckConstraint`)

### Функция `timer` (декоратор)

```python
def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"⏱ Время выполнения: {time.time() - start:.4f} сек")
        return result
    return wrapper
```

**Достоинства:**
- Простой и эффективный декоратор для замера времени
- Использует `time.time()` (достаточно для демонстрационных целей)

**Замечания:**
- Отсутствует `@wraps(func)` из `functools`, что приводит к потере метаданных функции (имени, docstring). Для учебного проекта это не критично, но в профессиональной разработке важно.

### Демонстрация N+1 проблемы (`demo_relationships`)

Это ключевая часть проекта, которая наглядно демонстрирует одну из самых частых проблем при работе с ORM.

```python
# LAZY LOADING (проблема)
authors = session.query(Author).all()
for author in authors:
    books_count = len(author.books)  # N дополнительных запросов

# EAGER LOADING (решение)
authors = session.query(Author).options(selectinload(Author.books)).all()
for author in authors:
    books_count = len(author.books)  # Данные уже загружены
```

**Достоинства:**
- Отличное наглядное сравнение двух подходов
- Использование `selectinload` (рекомендуемый способ для one-to-many)
- Код сопровождается выводом, но без указания количества запросов — стоило бы добавить логирование SQL

**Замечания:**
- В выводе не показывается реальное количество выполненных SQL-запросов. Для большей наглядности стоило включить `echo=True` в движке или использовать логирование.

### Демонстрация агрегации (`demo_aggregation`)

```python
stats = session.query(

    func.avg(Book.price).label('avg_price'),
    func.min(Book.price).label('min_price'),
    func.max(Book.price).label('max_price'),
    func.sum(Book.price).label('total_value')
).first()
```

**Достоинства:**
- Отличная демонстрация всех основных агрегатных функций
- Использование `.label()` для понятных имён в результате
- Дополнительная группировка по авторам через `join()` и `group_by()`

**Замечания:**
- В группировке используется `Author.name`, что может привести к проблемам при наличии одноимённых авторов. Правильнее было бы группировать по `Author.id`

### Демонстрация сложного запроса (`demo_complex_query`)

```python
subquery = session.query(
    Book.author_id,

expensive_books = session.query(
```

## Найденные недочёты и предложения по улучшению

### 1. Отсутствие контекстного менеджера для сессий

В коде сессии создаются и закрываются вручную:

```python
session = Session(engine)
# ... операции ...
session.close()
```

**Проблема:** При возникновении исключения между созданием и закрытием сессии, `session.close()` не будет вызван, что может привести к утечке соединений.

**Рекомендация:** Использовать контекстный менеджер:

```python
from contextlib import contextmanager

@contextmanager
def session_scope():
    session = Session(engine)
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()

# Использование
with session_scope() as session:
    author = Author(name="New Author")
    session.add(author)
# Автоматический commit и закрытие
```

### 2. Отсутствие обработки ошибок в демонстрационных функциях

Все демонстрационные функции предполагают, что данные уже существуют и операции выполняются успешно. При повторном запуске скрипта возникает ошибка уникальности email.

**Рекомендация:** Добавить очистку таблиц перед заполнением или проверку существования данных:

```python
def setup_test_data():

        # Создание тестовых данных
        # ...
```

### 4. Декоратор `timer` без `@wraps`

```python
def timer(func):
    def wrapper(*args, **kwargs):
        ...
    return wrapper  # Потеря метаданных
```

**Рекомендация:** Использовать `@wraps`:

```python
from functools import wraps

def timer(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"⏱ Время выполнения: {time.time() - start:.4f} сек")
        return result
    return wrapper
```

## Сравнение с альтернативными подходами

В документации приведена таблица сравнения SQLAlchemy с другими инструментами:

| Инструмент | Сильные стороны | Слабые стороны |
|------------|----------------|----------------|
| **SQLAlchemy** | Полнота, гибкость, производительность | Крутая кривая обучения |
